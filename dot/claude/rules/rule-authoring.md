## rules / CLAUDE.md の書き方

`.claude/rules/*.md` や CLAUDE.md に規約を書くときの指針。なぜこの形にするかという理由ごと
書くことで、agent が状況に応じて自分で判断できるようにする。

### 理由を併記する

規約には「なぜそうするか／何を防ぐか」を本文に書く。理由は規約本体と同じくらい重要で、
agent が「この理由は今の状況に当てはまるか」を判断する材料になる。理由の無い裸の禁止は、
当てはまらない状況でも盲目的に避けられ、かえって作業を妨げる。

### 縛りは理由付きのソフト指針にする

断定的なハード禁止（「絶対〜禁止」「必ず〜する」）は避け、「（一般に／特定状況で）〜しない」
という形にする。明記した理由が当てはまらない特定状況では、agent がその手段を選べる余地を
残すため。例: main への直 push は一般に避けるべきだが、その理由（並行作業の巻き込み等）が
無い状況では選択肢になりうる。「禁止だから使わない」で思考停止させない。

縛り切りたい（判断の余地を残したくない）必須事項は、rules ではなく lint / test / hook へ
昇格する。rules は順守が不完全な指針であり、確実に止めたいなら機械的な強制（hook 等）の
ほうが適しているため。

### 領域固有なら paths でスコープする

特定領域でしか効かない規約は frontmatter の `paths`（グロブ配列）でスコープし、その領域の
ファイルを触ったときだけロードされるようにする。常時ロードはコンテキストを圧迫するため、
文脈依存の規約は paths でスコープするほうがよい。常時必要な規約だけ `paths` 無しにする。

### 由来を残す

その規約が生まれた文脈・指摘を末尾に短く残す。後で「なぜこの規約があるのか」を辿れると、
不要になった規約の削除や見直しの判断がしやすい。
