# second-opinion

長時間の調査・修正でコンテキストが肥大化すると、誤った前提や hallucination が判断を
汚染し、「修正した→動作確認で解決しない」の空回りが増える。この skill は診断だけを
fresh context の sub-agent（最賢モデル）へ外部化する。sub-agent の価値はモデルの賢さ
以上に **fresh context** にある — 親のコンテキスト汚染から独立した判断が得られる。

## 不変条件（厳守）

- **sub-agent は診断のみ。** ファイル編集はさせない。修正の実施は診断を受けた親
  （またはユーザー判断）が行う。診断だけ外部化すれば「判断の誤り」には十分効き、
  実施まで委譲すると作業コンテキストの再構築コストが診断の利点を食い潰すため。
- **親（自分）の原因仮説はすべて「推測」として扱う。** 検証済みの実出力（事実）と
  物理的に分離して渡す。親が書く状況説明には汚染が混入しうるため、この分離が
  防波堤になり、事実と推測の乖離度そのものが劣化診断の材料にもなる。
- **仕切り直し勧告が出たら新たな修正を試みない。** ハンドオフ文書を書いて停止する。
  劣化した親に最後にやらせてよい仕事は「事実の書き出し」だけ。

## 手順

### 1. ブリーフ作成

現状を以下の 4 セクションで整理する。「事実」に書いてよいのは実際に観測した出力
（テスト結果・エラーメッセージ・コミット hash）だけ。解釈・信念が混じるものは
すべて「推測」へ移す。

```markdown
## 症状
<いま失敗している事象。再現コマンドとその実出力>

## 事実（検証済みの実出力のみ）
<試行履歴を時系列で。各回: 変更内容（コミット hash / diff の要点）→ 結果（実出力）>

## 推測（未検証の仮説）
<親が信じている原因仮説と、そう考えた根拠。「〜のはず」「〜と思われる」は全部ここ>

## 一次情報の所在
<関連ファイルパス・ログの場所・参照した docs の URL>
```

### 2. sub-agent 起動

Task tool で起動する（repo の subagent dispatch は Task に統一。pr-review-automerge
skill も同様）。Task は結果が返るまでブロックする同期実行なので、診断を待って受け取れる。
利用可能な最賢モデルを使う（この環境では `model: fable`。モデル名は環境に合わせて
読み替える）。プロンプトは以下の template にブリーフを埋め込む。

```
あなたは行き詰まった調査・修正タスクの診断医である。依頼元 agent は長いセッションで
コンテキストが肥大化しており、その判断は汚染されている可能性がある。あなたの価値は
fresh context にある。

<ここにブリーフ全文を貼る>

ブリーフの「推測」セクションは仮説にすぎない。信用せず、以下を自分でやり直してから
判断せよ:

1. 一次情報(対象コード・git log/diff・テスト出力・公式 docs)を自分で読み直す。
   必要なら再現コマンドやテストを自分で実行してよい。ただしファイル編集は禁止。
2. 「事実」と「推測」を突き合わせ、推測が一次情報と乖離していないか検証する。

最終出力として以下の 3 点を報告せよ:

1. 根本原因の仮説と根拠 — 一次情報の引用(ファイルパス:行、実出力)付きで。
2. 次のアプローチの提言 — 何をどの順で確認・修正すべきか具体的に。
3. 依頼元セッションの劣化診断 — 以下の兆候を検査し、「このまま継続可」か
   「仕切り直し推奨」かを明示して理由を述べよ:
   - ブリーフの推測が一次情報と乖離している(誤った前提の固定化)
   - 存在しない API・ファイル・挙動への言及(hallucination)
   - context rot / poisoning / distraction / confusion / clash に該当する兆候
     (過去の誤情報を引きずる、無関係な文脈に引きずられる、矛盾した前提の混在など)
```

### 3. report 提示と分岐

診断 report は**全文**をユーザーに提示する（要約で潰さない — 要約力自体が劣化して
いる可能性があるため）。

- **継続可** → 提言されたアプローチで作業を再開する。
- **仕切り直し推奨** → 手順 4 へ。

### 4. 仕切り直し（ハンドオフ）

新たな修正は試みない。以下だけ実施して停止する。

1. `~/.claude/handoffs/<repo名>/YYYY-MM-DD-<topic>.md` にハンドオフ文書を書く。
   `<repo名>` は作業ツリー toplevel の basename（`git rev-parse --show-toplevel` の
   basename。linked worktree ならその worktree の dir 名）。repo の外に置くのは、
   どの repo でも gitignore 整備なしに使え、git status を汚さず、誤 commit も
   起きないため。
2. 文書の内容: 症状・確定事項・試行済みアプローチと結果・**sub-agent 診断 report の
   全文貼り付け**。推測を書く場合は「推測」ラベル必須。
3. ユーザーに案内する:
   「新セッション（または `/clear` 後）でこのハンドオフ文書を読ませて再開してください:
   `~/.claude/handoffs/<repo名>/<file>.md`」

---

由来: コンテキスト肥大化したセッションで「修正→動作確認で解決しない」が繰り返され、
判断の誤りに agent 自身が気づけない問題への対策（2026-07-23 の brainstorm）。
機械検知（hook）は意図的にスコープ外 — まず自己申告 + 手動トリガーで効果を見る。
